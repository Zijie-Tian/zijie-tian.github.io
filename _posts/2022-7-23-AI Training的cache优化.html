Problem

在进行AI训练的过程中，需要将原始数据从storage系统中load进来之后再进行preprocessing，而preprocessing之后的实际数据往往远远小于从storage中读取的数据，这就导致了大量的I/O的浪费，产生大量的时间空闲。尤其时network-based的存储系统，会产生大量的I/O开销。

一般为了减少在线处理的开销，使用offline的模式先对所有数据进行处理，保存成中间数据，避免阻塞计算。但是，对于某些ML系统，比如图像方面的工作，会对数据进行数据增强，这个过程，如果要保存中间文件，会带来非常大的存储上的开销。

此外，针对于比较常用的SGD算法[2]，这里简单说一下这种算法：为了减少整体的loss function，例如：Q(w)=1n∑i=1nQi(w) ，这要求在每一个w处都要进行n次的运算，为了减少这一计算量，使用某一个sample的梯度进行梯度的优化：w:=w−η∇Qi(w)，在凸优化中可以证明其收敛性。

这种方法存在一个在每一个epoch中对于所有的sample都会遍历一遍，但是每一次的顺序不同，这个一般由伪随机的seed来决定。

 

Deep Learning Training task的I/O模式

Deep Learning Training的I/O模式有以下几个特点：

  1. 可以很好的共享，因为dataset存在head-heavy的特点。
  2. random access。
  3. Substitutability：每一个minibatch-SGD的过程，所用的sample的index并不是很重要，重要的是保证所有的samlple都可以被使用有且仅有一次到就可以了。
  4. Predictability：每一个step的随机序列仅仅取决于所用的random seed，完全是确定的。
  5. 大量的小文件I/O[6]，对于文件系统并不是很友好。

 

Solutions

这类问题有一些比较经典的解决方案：

  1. double-buffering：Pytorch，DIESEL[6]
  2. data sharding：data sharding[11]
  3. in-memory cache：Locality-aware caching[3]，tf.data[2]
  4. 对DLT的cache特征进行建模：tf.data[2]、NoPFS[10]

 

Trends

  1. 在NoPFS[10]中提到，设计一种generic、performance model-driven的system是一种趋势，应对多种复杂的I/O系统。
  2. 在NoPFS[10]中提到，DLT任务的节点，per-node的存储能力非常小，一般之后几十GB/node，需要进行distributed的一些优化。

 

 

Related Works

Quiver[1]

Deep Learning Training(DLT)任务主要存在两个比较重要的特性：

  1. DLT的dataset存在head-heavy的特性，也就是一小部分的数据被大量重复使用。
  2. DLT任务在训练时对同一数据集进行多个epoch的计算。

根据这两个特性，Quiver主要的解决思路就是在多个user之间安全地share相同的cache。Quiver主要由三个组件，Quiver Cache manager、Quiver client和Quiver server，整个系统结构如下图所示。

为了保证cache的Locality，Quiver在rack上尽可能使用相同switch的其他VM来共同进行cache，避免cross-rack的network traffic。

Quiver的cache是基于Content-hash-based的Read-Only File System来建立的[12]，对于cache的粒度，由对应的DLT Job来决定（有问题：这样not Eazy to Use），文中默认为一个sample为粒度。每一个job的container中都会维护一个digest文件，用于完成content,-based hash到cloud file path的映射。

为了在reuse cache的同时做好data之间的isolation，Quiver利用了content-based的hash的稀疏性，避免可以通过generate hash的方式泄露数据。（Quiver是可以选择cache的粒度的，如果这里使用dataset粒度的cache，那就可以彻底在不同的dataset之间进行隔离。）

Quiver cache manager用于协调在Quiver client和Quiver server的cache的调度，包括相关的逐出策略等等。

在data fetch的过程中，Quiver使用了Substitude hits的机制，这种机制主要为了避免一个mini-batch中仅仅只有一小部分在cache中，在每一个mini-batch中，Quiver每一次通常会取一个长出mini-batch size数倍的一个序列（比如10x），并将所有没有在cache中的sample标为pending，在这里面挑出一个mini-batch进行一个pass，这样并不会影响SGD的性质，因为保证了所有sample都遍历一遍的条件。

上面这种做法存在一个非常大的问题，就是cache在比较小的时候，这个性能不会很好，因此需要一种比较好的cache的逐出算法。Quiver的逐出算法采用类似double-buffering的方法，在Pytorch[5]也有应用，大概的流程为：

  1. 将dataset分为两个chunk，当由多个job对其中一个chunk进行access的时候，另一个chunk加载进入cache。
  2. 只有当第二个chunk完成加载并且第一个chunk中不再有job进行hit的时候，开始在第一个chunk的空间中load第三个chunk。

这里有一个问题：如果在一个chunk上，存在一个job拖慢了整个cache的逐出怎么办？Quiver的的处理方法就是直接不管，将chunk逐出，因为Quiver认为这种pre-minibatch时间比较长的任务，不会是一个I/O-bound的任务，所以，不用对其的I/O进行cache方面的优化。

在进行cache-missing的处理的时候，为了提高效率，Quiver使用多个进程进行fetch，但是，并没有一个类似于leader的进程来安排具体每个进程处理cache-missing，将不同的sample shard到不同的fetch进程，为了避免不同fetch进程之间出现overlap，每一次取的时候，都会check本地cache是否exist。（这里比较有问题，这种方式其实并不好，因为不能保证所有fetch都会成功，但这问题不大，我觉得还是memcached的机制更好一点。）

总体来说，Quiver确实是一个非常好的系统，为后面的工作给了一个基础。

 

Hoard[8]

Hoard对DLT的I/O特征分析主要有4点：

  1. DLT的dataset往往非常大，单个节点依靠Linux的buffer cache[9]根本不够，因此，需要在多个节点做分布式cache。
  2. 对于DLT来说，dataset中一部分cache不命中，会影响整个dataset的性能。（因为每一个epoch都会random sample）。因此，需要以dataset为粒度进行cache。
  3. 调度和cache需要协同。（？）
  4. cache需要使用标准的POSIX接口。（？）

如上图所示，Hoard对上由Hoard API Server暴露POSIX API给上面的DLT任务，在Scheduling layer使用Kubernetes custom resources controller进行控制，主要由两种scheduler，一种是DL Job的调度，另一种是Dataset的调度，将不同的dataset和DL job绑定。dataset的调度actions将传给下面的dataset manager，一旦distributed cache layer将dataset准备好，就对上暴露为一个persistent volume，在distributed cache layer中，利用分布式文件系统，实现在多个节点上的cache。

Distributed File System的选择：

  1. GlusterFS：不能原生支持cache，需要修改代码，比较麻烦。
  2. Alluxio：不能选择在某些节点上进行cache，只能支持在所有节点上进行cache。
  3. Spectrum Scale：可以用作为Hoard的Distributed的cache。

在上层使用Active File Management（AFM），将分布式文件系统作为cache来使用。下图为实验中使用的cluster的结构图。

 

DIESEL[6]

DIESEL不同于其他系统的点在于：DIESEL是一个storage-layer和cache-layer的co-design的一个系统。比较经典的Quiver[1]是一个工作在cache-layer的系统。DIESEL针对于DLT任务的大量小文件I/O进行了优化，整个系统的结构如下图所示。

DIESEL中，使用object-storage对于dataset进行存储，并且使用Distributed KV存储存储所要用到的metadata，使用etcd存储整个系统的configure。其中，DELSEL server将所有对于小文件的请求进行merge和sort，按照chunk为单位进行操作。DLCMD是一个比较简单的命令操作，类似于Amazon的s3cmd，用户可以通过这个DLCMD对DLESEL server上的文件进行读写操作。

对于DIESEL的cache机制，主要有两点：

  1. 为了在节点之间分布式cache所有的dataset，在某一个node上的job出现cache missing的时候，需要保证可以从remote的cache中拿到，但是这样会导致所有的job之间需要建立n ∗ (n-1)个链接，DIESEL以job为基本的粒度，在每一个node之内选出一个master job作为和其他node通信的进程，这样仅仅只需要建立p ∗ (n-1)个链接，减少了开销。
  2. 为了进一步提升cache的性能，DIESEL利用了DLT的多数会利用之前pretrained model的特性，在加载pretrained model的时候，同时在background从server端获取数据，尽可能减少第一个epoch cache造成 的latency。同时，DIESEL也支持on-demand的cache，这样在所有的epoch都尽可能让cache miss导致的latency相差不多。

Data Sharding[11]

Data sharding的主要思路有两个方面，一方面从系统进行优化，另一方面从Deep Learning的层面进行优化。

System层面：

  1. 由于大规模的数据集往往非常大，naive的对于数据集进行分割读取的方法会导致每一个data file有可能被多个node读取，因此设计了一个新的Distributed data staging系统。先将dataset分隔为多个段，不同的段分到不同的node上，并且node内部使用多进程或者多线程的方式进一步提升读取文件的并行性。不过存在一些超算系统没有local storage，需要使用tmpfs，会带来一定的限制。
  2. Hierarchical all-reduce：基于数据并行的方法有可能会导致几个问题，首先第一个问题就是一个node中的多个rank的问题，随着node的数量的增加，会导致并行效率变得非常差。第二个问题就是不同的Collective Communication方法会有不同的性质，主要体现在MPI库原生的通信会有更好的性能，但是占用的带宽更大，NCCL的Collective Communication方式会由于ring的实现模式导致性能较差，但是却有着很好的bandwidth。文中的方法倾向于将二者结合起来，使用混合的Collective Communication的方式，在node内部使用NCCL的通信方式，在节点之间使用MPI的通信方式，最大地发挥performance和bandwidth。

Deep Learning层面：

  1. 这篇文章是专门针对于climate analytics来写的，对应于图像处理的背景分割技术，对于背景分割技术，在一张图片中，背景的面积要远远大于前景的面积，如果每一个pixel在计算loss函数的时候相同权重，会导致对于背景类别（占比比较大的）的accuracy会比较高，文中设计了一种weighted loss进行解决。
  2. LARC和多通道分割，比较偏AI的优化。
  3. 使用Gradient lag技巧提高并行度，在进行gradient update的时候，使用上一步的gradients进行计算，尽可能提高并行性。

Locality-aware loading[3]

一方面，主要针对于data load过程中的并行性进行优化，在数据加载阶段，使用mult-processing对整个dataload的过程进行优化，提升整个load的throughput，同时针对于每一个单独的数据的preproccessing过程，进行mult-threading的优化，可以进一步减少时间开销，但是由于Python的GIL的问题，这种优化只有在正确释放锁的时候才能达到比较好的效果。

另一方面，Locality-aware loading在cache的方面进行优化，一般来讲，在Data-Parallell的过程中，多个learner之间的数据和训练的过程如下图所示。

可以在DRAM中使用software cache，减少对于storage的访问，从而减少因为dataload而导致的wait dataload的过程。

此外，文章内还使用了distributed cache进行了优化，用这种方式，对于每一个sample的情况就有在本地cache中命中、在remote的cache中命中和在storage中找，那么这样的每一个sample的I/O的时间可以用下面的公式进行计算。

(1) Sample I/O time =(1−α)⋅DR⏟Storage system +α⋅DRc⏟Remote caches ⋅p−1p⏟Local cache miss 

其中R是从storage的I/O rate，Rc是从远端的cache到本地的速率，文中假设Rc>R， 因为假设在不同node之间存在高带宽的P2P设备。

要在minibatch-SGD的前提下实现较好的并行，就要避免不同的节点之间的sample的量不同，并尽可能保证尽可能多的dataload在local cache中实现。根据传统的minibatch-SGD，locality-aware loading利用了节点之间的高速传输设备，在不同的节点之间做load-balance。一种比较常见的情况如下图所示：

NoPFS[10]

Near-optimal PreFetching System（NoPFS），NoPFS和之前的经典系统的对比，见下图：

在modern DLT场景中，I/O subsystem变得越来越复杂多样，例如越来越多的趋势倾向于使用DAOS和Rabbit，这种不同对于DLT的性能会受到影响，设计一种比较generic、performance model-driven的system会是一个比较好的趋势。下图为两种不同的I/O subsystems。

从上图不难看出，在传统的system中，一般per-node的storage往往会比较小，例如，Piz Daint--64GB/node，Fugaku--32GB/node，这些系统往往在进行DLT任务的时候还会在storage中保存checkpoint等，会导致local storage进一步不够用。因此，对于多种不同的I/O system的generic非常重要。NoPFS在存储上使用的是on-node和distributed memory结合的一种cache。

NoPFS试图去找到一种接近最优的一种cache逐出算法，根据文中的说法，最优的算法应当满足：

  1. Optimal prefetching：在未来R个sample内要访问到的sample，需要及时进行prefetch。
  2. Optimal replacement：在未来最久要被访问到的sample，需要及时逐出。
  3. Do no harm：不会出现为了prefetch，逐出将要被access的sample。
  4. First opportunity：不要进行prefetch-and-relpace，如果之前进行过一样的操作。[?]

NoPFS严格保证其中的1，近似保证其他三个，实现一种Near-optimal的状态。NoPFS首先对sample的access pattern进行建模，不难发现，在E个epoch中，可以确定地是每一个sample一定都会被access E次，但是同一个sample并不能在多个worker中，并不能确定其地access次数，可以用一个分布来进行表示，设一个sample在不同的E个epoch中满足Bernoulli分布，Xe∼ Bernoulli (1N)，那么，在E个epoch中，每一个sample的access frequency满足 X=∑e=1EXe，也就是：

(2)P(X>(1+δ)μ)=∑k=⌈(1+δ)μ⌉K(Ek)(1N)k(N−1N)K−k.
